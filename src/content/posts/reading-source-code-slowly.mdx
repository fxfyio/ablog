我以前读源码喜欢“从头到尾”，后来发现效率不高。

现在的做法是先带着问题进去，例如：

- 某个 API 为什么这样设计？
- 一个错误是在哪一层被转换的？
- 配置项最终在哪里生效？

## 更实用的阅读路径

1. 先找到入口
2. 跟一条真实调用链
3. 在关键节点打断点或加日志
4. 最后再回看整体结构

这种读法看起来慢，但记得更牢。
